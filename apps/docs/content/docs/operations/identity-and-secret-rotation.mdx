## Forced MFA Reset After Account Recovery

Qryvanta does not yet expose an admin HTTP endpoint for forced MFA reset.
The emergency operator path is therefore a controlled database change plus session revocation.

Use this only after a human identity-verification process outside the product has succeeded.

### Preconditions

1. Confirm the recovery request in your support system with at least two independent proofs of identity.
2. Confirm the target user id and primary email address.
3. Notify the user that all active sessions will be revoked and MFA re-enrollment will be required.

### Emergency Reset Procedure

Run the following transaction against the primary application database, replacing `<user_uuid>` with the recovered user's id:

```sql
BEGIN;

UPDATE users
SET totp_enabled = FALSE,
    totp_secret_enc = NULL,
    recovery_codes_hash = NULL,
    totp_pending_secret_enc = NULL,
    recovery_codes_pending_hash = NULL,
    auth_sessions_revoked_after = now(),
    updated_at = now()
WHERE id = '<user_uuid>'::uuid;

COMMIT;
```

### Post-Reset Actions

1. Instruct the user to complete password reset before next login if the recovery event involved possible credential compromise.
2. Require the user to sign in again and re-enroll MFA from `/auth/mfa/totp/enroll` and `/auth/mfa/totp/confirm`.
3. Verify the old session cookie is rejected and the recovered account shows `totp_enabled = false` before re-enrollment.
4. Record the ticket id, operator, timestamp, and reason in your incident system.

### Validation Queries

```sql
SELECT id,
       email,
       totp_enabled,
       auth_sessions_revoked_after,
       updated_at
FROM users
WHERE id = '<user_uuid>'::uuid;
```

Expected state:

- `totp_enabled = false`
- `auth_sessions_revoked_after` set to the recovery timestamp
- TOTP and recovery-code columns cleared

## Secret Rotation Policy

Minimum policy:

1. Rotate `AUTH_BOOTSTRAP_TOKEN` and `WORKER_SHARED_SECRET` on a fixed schedule and after any incident involving leaked configuration or infrastructure access.
2. Rotate `SESSION_SECRET` during a planned maintenance window because all browser sessions will be invalidated.
3. Rotate `TOTP_ENCRYPTION_KEY` only with a coordinated MFA re-enrollment campaign because current Qryvanta releases do not support seamless multi-key decryption.

Keep all secret values unique per environment.
Never promote the same value from local to staging or from staging to production.
When runtime config is sourced from `*_FILE` or `*_SECRET_REF`, rotation means updating the referenced file or manager entry and then restarting the affected API and worker processes.
After each rotation, refresh the `print-secret-fingerprints` output for that environment and distribute the updated fingerprint records to the other environments' `SECRET_REUSE_GUARD_FINGERPRINTS` lists.

## Bootstrap Token Rotation

Secret:

- `AUTH_BOOTSTRAP_TOKEN`

Procedure:

1. Generate a new high-entropy token in your secret manager.
2. Update `AUTH_BOOTSTRAP_TOKEN` for every API replica.
3. Restart or redeploy API processes.
4. Validate:
   - `GET /health` returns `ready=true`
   - Existing authenticated sessions continue to work
   - Old bootstrap token is rejected

Operational effect:

- No authenticated user-session impact
- Any automation or internal setup flow using the old bootstrap token fails immediately

## Worker Shared Secret Rotation

Secret:

- `WORKER_SHARED_SECRET`

Qryvanta currently uses a single active worker secret.
There is no dual-key grace period, so rotation must be coordinated.

Procedure:

1. Pause queued workflow processing or accept a short worker interruption window.
2. Generate a new `WORKER_SHARED_SECRET`.
3. Update the secret for API and every `qryvanta-worker` deployment.
4. Redeploy API and workers in one maintenance window.
5. Validate:
   - `GET /health` returns `ready=true`
   - Worker claim loops resume successfully
   - No sustained `401` responses on internal worker routes

Operational effect:

- Existing browser sessions are unaffected
- In-flight queued work may be delayed during the restart window

## Session Secret Rotation

Secret:

- `SESSION_SECRET`

Procedure:

1. Schedule a maintenance window because all authenticated browser sessions will be invalidated.
2. Generate a new `SESSION_SECRET` in your secret manager.
3. Deploy the new value to every API replica at the same time.
4. Restart API processes.
5. Validate:
   - `GET /health` returns `ready=true`
   - Old session cookies are rejected
   - New login creates fresh authenticated sessions successfully

Operational effect:

- All users must authenticate again
- Existing server-side session rows may remain in storage until expiry, but old signed cookies stop working

## TOTP Encryption Key Rotation

Secret:

- `TOTP_ENCRYPTION_KEY`

Current platform limitation:

- Qryvanta supports exactly one active MFA encryption key at runtime.
- Existing stored TOTP secrets cannot be transparently re-encrypted during normal operation.
- If `TOTP_ENCRYPTION_MODE=aws_kms_envelope`, new enrollments use AWS KMS-wrapped per-secret data keys, but older static-encrypted rows still require `TOTP_ENCRYPTION_KEY` until users re-enroll or are reset.

In `static` mode, that means rotation requires a forced MFA reset campaign for all enrolled users.

If `TOTP_ENCRYPTION_MODE=aws_kms_envelope`, `TOTP_KMS_KEY_ID` controls wrapping for new enrollments only. Existing wrapped rows remain bound to the KMS key that originally encrypted their per-secret data keys, and older static rows still require `TOTP_ENCRYPTION_KEY` until they are reset or re-enrolled.

Procedure:

1. Export the set of users with `totp_enabled = true` for the target environment.
2. Announce an MFA re-enrollment window to affected users.
3. For each affected user, clear MFA state and revoke sessions using the forced-reset SQL transaction from this page.
4. Generate a new 64-hex-character `TOTP_ENCRYPTION_KEY`.
5. Update API configuration and restart all API replicas.
6. Validate:
   - API startup succeeds
   - Newly enrolled MFA secrets can be confirmed
   - Previously enrolled users are prompted to re-enroll MFA

Discovery query:

```sql
SELECT id, email
FROM users
WHERE totp_enabled = TRUE
ORDER BY email;
```

Operational effect:

- Every user previously enrolled in MFA must re-enroll
- Recovery codes are invalidated and must be regenerated during re-enrollment

AWS KMS envelope mode note:

- Moving the config from one `TOTP_KMS_KEY_ID` to another does not automatically rewrite older wrapped MFA rows.
- Keep the previous AWS KMS key available for decrypt until every affected user has re-enrolled MFA or has been reset.

## Post-Rotation Checklist

1. Confirm health and login flows in the target environment.
2. Confirm audit and incident records were updated with rotation timestamp and operator.
3. Revoke or delete superseded secret material from the old storage location.
4. Update any break-glass documentation that references rotated values.
5. For `WORKER_SHARED_SECRET` and `SESSION_SECRET`, monitor logs for elevated unauthorized responses during the first restart window.
