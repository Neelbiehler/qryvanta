Qryvanta ships a reproducible `PERF-04` benchmark suite under `benchmarks/k6`.

The suite focuses on production-relevant API paths:

- Workspace surface reads (`GET /api/v1/workspace/apps`)
- Runtime query load (`POST /api/v1/workspace/apps/sales_hub/entities/deal/records/query`)
- Workflow execution load (`POST /api/v1/workflows/deal_created_notify/execute`)

## Prerequisites

1. Start infrastructure (`pnpm infra:up`).
2. Start API (`pnpm dev:api`) on `127.0.0.1:3001`.
3. Keep default seeded credentials unless overridden:
   - `admin@qryvanta.local`
   - `admin`

The runner seeds deterministic CRM/ERP fixture data automatically before each run.

Nightly/manual benchmark automation is defined in `.github/workflows/performance-benchmark.yml`.

## Run Benchmarks

Run mixed profile:

```bash
pnpm perf:benchmark
```

Run a specific profile:

```bash
pnpm perf:benchmark -- --profile runtime --duration 120s
```

Equivalent `just` command:

```bash
just perf-benchmark mixed 120s
```

Each run writes a JSON summary to `benchmarks/results/perf-<profile>-<timestamp>.json`.

## Profiles

- `workspace`: reads workspace app metadata route only.
- `runtime`: stresses runtime record query route only.
- `workflow`: stresses workflow execute route only.
- `mixed`: runs all three scenarios concurrently.

## Tuning Knobs

Override scenario rates and VU limits with environment variables:

- `BENCHMARK_WORKSPACE_RATE` (default `15`)
- `BENCHMARK_RUNTIME_QUERY_RATE` (default `20`)
- `BENCHMARK_WORKFLOW_EXEC_RATE` (default `5`)
- `BENCHMARK_PREALLOCATED_VUS` (default `20`)
- `BENCHMARK_MAX_VUS` (default `80`)
- `BENCHMARK_DURATION` (default `90s`)

Pair benchmark rates with API-side backpressure settings when tuning:

- `RUNTIME_QUERY_MAX_LIMIT`
- `RUNTIME_QUERY_MAX_IN_FLIGHT`
- `WORKFLOW_BURST_MAX_IN_FLIGHT`

Auth and endpoint overrides:

- `QRYVANTA_API_BASE_URL`
- `QRYVANTA_FRONTEND_URL`
- `QRYVANTA_BENCH_EMAIL`
- `QRYVANTA_BENCH_PASSWORD`
- `QRYVANTA_BENCH_BOOTSTRAP_TOKEN` (recommended for repeated benchmark runs)
- `QRYVANTA_BENCH_BOOTSTRAP_SUBJECT`

## Reproducibility Notes

- The runner uses local `k6` when available.
- If local `k6` is missing, it falls back to `grafana/k6:0.49.0` via Docker.
- Bootstrap auth is opt-in via `QRYVANTA_BENCH_BOOTSTRAP_TOKEN`; use a subject with the required app/entity/workflow permissions.
- Seed data is regenerated by default (`pnpm dev:seed`) to keep benchmark payloads stable.
- Use `--skip-seed` only when intentionally benchmarking an existing dataset snapshot.
