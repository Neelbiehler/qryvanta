## Draft And Publish Operations

Workflows are now draft-first:

- `POST /api/workflows` saves the editable draft definition.
- `POST /api/workflows/{workflow_logical_name}/publish` snapshots that draft as the next immutable published version and makes it active.
- `POST /api/workflows/{workflow_logical_name}/disable` stops future dispatch while keeping the latest published version for history and rollback-oriented operations.

Operational notes:

- Draft changes do not execute until a publish happens.
- Manual execution and trigger dispatch resolve the current published version only.
- Each workflow run stores the published `workflow_version` used at dispatch time, so queued workers and step retry keep using the same snapshot even if a newer draft or publish exists later.

## Trigger Coverage

Backend runtime supports these trigger types:

- `manual`
- `runtime_record_created`
- `runtime_record_updated`
- `runtime_record_deleted`
- `schedule_tick`
- `webhook_received`
- `form_submitted`
- `inbound_email_received`
- `approval_event_received`

`runtime_record_*` triggers are written to a transactional outbox during runtime API and workspace record mutations. Inline mode drains that outbox immediately after the write path returns. Queued mode drains it through the authenticated worker control loop before workers claim execution jobs.

Current chaining rule:

- Runtime writes emitted by workflow execution do not trigger additional workflows.
- This is intentional for now so platform writes cannot recurse until bounded depth and loop-detection semantics are available.

## Schedule Trigger Operations

Queued workers now include a built-in scheduler pass before runtime-event drain and job claim. For each enabled `schedule_tick` workflow trigger, the worker derives the current due UTC slot, claims it in `workflow_schedule_ticks`, and dispatches the workflow exactly once for that slot.

Built-in slot formats:

- `hourly`
- `daily`
- `daily_utc_HHMM`
- `weekday_utc_HHMM`

The protected API dispatch endpoint remains available for manual backfill, replay, or custom scheduler integration:

- `POST /api/workflows/triggers/schedule/dispatch`

Request body:

```json
{
  "schedule_key": "hourly",
  "payload": {
    "tick_at": "2026-03-01T00:00:00+00:00",
    "timezone": "UTC",
    "source": "scheduler-primary"
  }
}
```

Notes:

- Built-in scheduling currently evaluates UTC schedule keys only.
- Endpoint requires authenticated protected API access.
- Use stable schedule keys like `hourly`, `daily`, `daily_utc_0900`, or `weekday_utc_1730`.
- `payload.tick_at` (or legacy alias `payload.tick`) should be RFC3339; runtime normalizes to UTC.
- Workflow trigger payload includes deterministic schedule fields:
  - `tick_at_utc`
  - `tick_source` (`payload` or `system_clock`)
  - `timezone`
  - `clock_skew_seconds`
  - `clock_skew_tolerance_seconds`
  - `clock_skew_within_tolerance`

## Webhook Trigger Operations

Native webhook ingress is now available at:

- `POST /api/public/workflows/webhooks/{tenant_id}/{webhook_key}`

Request body:

```json
{
  "customer_id": "cust-1",
  "event_type": "customer.created"
}
```

Runtime wraps the inbound request into a workflow trigger payload with:

- `event` = `webhook_received`
- `webhook_key`
- `request.method`
- `request.headers`
- `request.query`
- `payload`
- `data`

Notes:

- The endpoint is tenant-scoped by path and workflow-scoped by `webhook_key`.
- It is intentionally public so external systems can post directly.
- Inline credential-bearing outbound headers (`authorization`, `x-api-key`, `cookie`, and similar token headers) are blocked at workflow publish time unless they are configured through `header_secret_refs`.
- `header_secret_refs` resolve supported secret-manager references (`op://`, `aws-sm://`, `aws-ssm://`, `vault://`, `gcp-sm://`) at dispatch time. Maker now exposes typed secret-header presets, `Authorization` format selectors (`Raw`, `Bearer`, `Basic`), and provider-aware secret-ref builders for common outbound auth headers, but provider-specific signing flows are still not implemented.
- In Maker, webhook payloads now use typed field-row editors for common object payloads; nested subdocuments still use per-field `JSON` values when needed.

## Outbound Integration Actions

Workflow runtime dispatches these native integration actions through external adapters:

- `http_request`
- `webhook`
- `send_email`

### HTTP and Webhook Dispatch Behavior

- Runtime sends outbound requests using the configured HTTP client.
- Retries transient failures (5xx and `429`) with bounded backoff.
- Treats non-retryable 4xx responses as terminal validation failures.
- Sets headers for idempotency and traceability:
  - `Idempotency-Key`
  - `X-Qryvanta-Workflow-Run`
  - `X-Qryvanta-Workflow-Step`

### Email Dispatch Behavior

- `send_email` actions use the platform email provider.
- `EMAIL_PROVIDER=smtp` requires valid `SMTP_*` environment values.
- Worker process falls back to console email if SMTP config is invalid; treat this as a misconfiguration in production and alert immediately.

## Native Platform Actions

Workflow runtime also supports native platform-side actions without disguising them as workflow-authored create-record steps:

- `update_runtime_record`
- `delete_runtime_record`
- `assign_owner`
- `approval_request`
- `delay`

Operational notes:

- `update_runtime_record` and `delete_runtime_record` execute through unchecked metadata-service writes inside workflow execution.
- `assign_owner` currently persists a `record_assignment` platform record internally.
- `approval_request` currently persists an `approval_request` platform record internally.
- `delay` is a bounded in-worker pause step. It is not yet a durable scheduler-backed wait primitive, so reserve it for short waits and use queued workers in production.

Maker editor notes:

- Create-record, update-record, webhook-payload, and approval-payload steps now use typed field-row editors instead of whole-object JSON textareas.
- Condition steps now use typed comparison controls (`Text`, `Number`, `Boolean`, `Null`, `JSON`) instead of a raw JSON value field.
- The test tab now provides typed sample trigger payload editing plus trigger-aware sample presets, so operators can dry-run published workflows without hand-writing a full payload blob first.
- Common object-shaped steps now render a local JSON preview card in the inspector to show the effective payload/request shape before execution.
- HTTP request steps now support `No body`, typed object-body editing, typed array/scalar bodies, or raw JSON mode. Raw JSON is now reserved for advanced nested custom payloads.

## Idempotency and Retries

Each external step derives a stable key:

- `<run_id>:<step_path>`

This key remains stable for run retries and step retries, so downstream services can deduplicate side effects.

For production endpoints:

1. Persist idempotency keys for a bounded retention window.
2. Return previously computed results for duplicate keys.
3. Keep handlers safe for at-least-once delivery.

## Token Usage in Action Payloads

Use workflow tokens to build outbound payloads:

- `{{trigger.payload.*}}`
- `{{trigger.*}}`
- `{{run.id}}`
- `{{run.attempt}}`
- `{{now.iso}}`

Prefer including `{{run.id}}` and business record ids in external payloads for supportability.

## Deterministic Replay

Use replay output to reconstruct one workflow run from persisted execution history:

- `GET /api/workflows/{workflow_logical_name}/runs/{run_id}/replay`

Response includes:

- Run snapshot (`run`)
- Ordered attempts (`attempts`)
- Flattened deterministic timeline (`timeline`)
- Stable checksum (`checksum_sha256`) for replay integrity checks

Operational guidance:

1. Capture `checksum_sha256` in incident tickets when coordinating across teams.
2. Compare checksums before/after export/import validation to confirm replay-equivalent history.
3. Use `timeline` sequence values for deterministic step ordering during postmortems.

## Production Readiness Checklist

1. Run API and worker in queued mode for non-trivial automation throughput.
2. Verify SMTP credentials and sender domain alignment before enabling `send_email` actions.
3. Ensure outbound firewall rules allow required webhook/API targets.
4. Confirm downstream systems enforce idempotency keys.
5. Monitor runtime trigger outbox drain health, schedule tick drain health, workflow queue depth, and worker heartbeat freshness.
6. Monitor workflow run retries and dead letters.

## Incident Triage

When automation incidents are reported:

1. Check workflow run history and step traces for failed step path and error details.
2. Validate downstream endpoint status and response patterns (`429`, `5xx`, auth errors).
3. Confirm worker heartbeat and queue depth are healthy.
4. Re-run failed steps only after downstream or configuration issues are fixed.
5. Capture run id and idempotency key in incident records for cross-system correlation.
