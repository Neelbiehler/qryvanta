Qryvanta is easier to reason about once you stop treating it like a hand-built CRUD app.
The runtime follows published metadata, tenant-scoped policy, and execution mode.

<DocCallout title="Start here when behavior is confusing" tone="tip">
  Most user-facing surprises in Qryvanta come from one of four places: unpublished draft changes, missing app bindings, role access, or workflow execution mode.
</DocCallout>

<DocSummary>
  <DocSummaryItem label="Use this section when">
    Something in the product feels inconsistent and you need the rule behind it.
  </DocSummaryItem>
  <DocSummaryItem label="Check first">
    Publish state, app bindings, role access, and workflow execution mode.
  </DocSummaryItem>
  <DocSummaryItem label="Common mistake">
    Assuming a draft change should already appear in Worker Apps.
  </DocSummaryItem>
</DocSummary>

## The Mental Model

<DocCardGrid columns={3}>
  <DocCard href="/docs/concepts/platform-architecture" eyebrow="System map" title="See how the whole project fits together">
    Use the architecture guide for the monorepo layout, runtime request flow, publish flow, and Rust-first API contract model.
  </DocCard>
  <DocCard href="/docs/concepts/metadata-and-publishing" eyebrow="Contract boundary" title="Draft is not runtime">
    Draft metadata can change freely. Runtime behavior changes only after publish succeeds.
  </DocCard>
  <DocCard href="/docs/concepts/records-and-views" eyebrow="Runtime model" title="Records run on published schema">
    Worker-facing list and form behavior follows the active published model, not whatever is sitting in draft.
  </DocCard>
  <DocCard href="/docs/concepts/automation-basics" eyebrow="Execution model" title="Automation reacts to events and schedules">
    Workflow runs can be triggered by runtime events, native schedules, webhooks, forms, inbound email, and approvals.
  </DocCard>
</DocCardGrid>

## What To Check First

<Checklist>
  <li>The expected metadata version has actually been published</li>
  <li>The entity is bound into the app navigation used by the affected role</li>
  <li>The role has the access needed for the runtime action</li>
  <li>The workflow execution mode and worker runtime match the way the tenant is configured</li>
</Checklist>
