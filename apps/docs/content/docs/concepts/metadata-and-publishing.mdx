Makers edit metadata in draft mode.
Runtime behavior changes only after publish.
That boundary is one of the main reasons Qryvanta stays predictable for runtime users.

## Draft to Runtime

1. Define or update entities and fields.
2. Update forms, views, and app navigation.
3. Run publish checks.
4. Publish selected scopes.
5. Validate the result in Worker Apps.

## What Publish Checks Catch

- Relation dependencies
- Missing field references in forms and views
- Published-schema compatibility breaks
- Invalid sitemap or app bindings
- App permission and binding issues that block runtime access

## What Publish Protects

Publish is not just a release button.
It is the place where Qryvanta checks whether a draft is safe to become the active contract for runtime APIs, forms, views, and workflows.

## Compatibility Gates

When an entity already has a published schema, publish checks block breaking contract changes.

Blocked examples:

- Removing a previously published field.
- Tightening `optional -> required` on a published field.
- Tightening `non-unique -> unique` on a published field.
- Changing field type, relation target, or option-set reference for a published field.
- Adding stricter text or numeric constraints after publish (`max_length`, `min_value`, `max_value`).
- Removing previously published option values from an option set.

Safe pattern:

1. Add new fields or option values first.
2. Migrate runtime records and forms/views to the new shape.
3. Keep old published contracts until rollout is complete.
4. Publish compatibility-safe changes in small steps.

## Common Rule

If users report missing fields or old layouts, verify the latest published version first.

Draft changes do not affect runtime pages until publish completes.

## Safe Change Pattern

1. Create or edit metadata in a draft.
2. Run checks and fix blocking issues.
3. Publish during a defined change window.
4. Verify with a pilot worker user before broad rollout.
