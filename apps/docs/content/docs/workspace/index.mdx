Qryvanta splits daily work into three surfaces.
Admin Center handles access and tenant controls.
Maker Center changes metadata.
Worker Apps runs the published product.

  <AudienceTag>Operators validating permissions</AudienceTag>

## Surface Model

<DocCardGrid columns={3}>
  <DocCard href="/docs/workspace/admin-center" eyebrow="Control plane" title="Admin Center">
    Manage roles, memberships, invites, registration behavior, and audit visibility for the tenant.
  </DocCard>
  <DocCard href="/docs/workspace/maker-center" eyebrow="Build plane" title="Maker Center">
    Define metadata, configure forms and views, bind apps, run publish checks, and publish changes.
  </DocCard>
  <DocCard href="/docs/workspace/worker-apps" eyebrow="Runtime plane" title="Worker Apps">
    Run the day-to-day record work that published metadata makes available to users.
  </DocCard>
</DocCardGrid>

## Recommended Team Flow

<Checklist>
  <li>Admin sets up access, roles, and invite policy</li>
  <li>Maker defines the entity model, forms, views, and app navigation</li>
  <li>Maker runs publish checks and publishes the change</li>
  <li>Worker users use the new runtime path in Worker Apps</li>
  <li>Admin reviews access and audit events when needed</li>
</Checklist>

## Practical Rule

<DocCallout title="Fix the right layer" tone="note">
  If a user cannot reach a page, start with roles and app bindings.
  If a page loads with the wrong fields or layout, start with published metadata.
  If a workflow or runtime side effect is missing, check the publish state and execution mode before changing code.
</DocCallout>
